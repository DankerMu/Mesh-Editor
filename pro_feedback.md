你说得对。我前面反复用“MVP”容易造成误解。更准确地说：

**你现在的方案不是“只适合做 MVP”，而是已经形成了最终成品的核心业务架构。**
但它目前更偏“可开发、可落地的核心系统方案”，如果要作为**最终业务成品**，还需要补齐一批“产品化、业务化、运维化、复盘资产化”的能力。

当前方案已经明确了主边界：固定 0.05° 等经纬度网格、24h/48h/168h 累计降水与相态订正、EC IFS 多要素只进入复盘绘图，不进入主编辑流程；同时也已经定义了 `tp(end_lead)-tp(start_lead)` 差分、`edit_session`、`preview_id`、7 张表、绘图三层接入架构等关键工程机制。

所以最终成品不应该推翻当前设计，而应该在当前设计上补成下面这个形态：

> **业务编辑系统 + 复盘归档系统 + 质量控制系统 + 审核发布系统 + 运维监控系统 + 历史分析系统。**

# 1. 最终成品的定位应当升级

现在方案里的核心闭环是：

```text
选择累计窗口
→ 显示累计降水和相态
→ 区域化修改量级或相态
→ 预览和应用
→ 保存订正前后格点
→ 生成差异和修改掩膜
→ 记录操作日志
→ 调用绘图代码
→ 生成并归档复盘数据包
```

这个闭环是**编辑与复盘闭环**，已经很扎实。

最终成品还应再补两个闭环：

```text
业务发布闭环：
编辑 → 审核 → 正式发布 → 产品归档 → 可追溯回滚

质量评估闭环：
编辑结果 → 复盘图 → 实况/人工复核 → 评分/标签 → 历史样本库
```

也就是说，最终产品不是简单“能改图、能出图”，而是：

```text
能值班使用
能审核发布
能出错回退
能复盘追责
能批量分析
能长期沉淀样本
```

# 2. 最终成品建议分成 8 个能力域

我建议最终成品按 8 个模块完善。

---

## 一、数据接入与产品窗口管理

当前方案已经定义了 `tp/pr/ptype` 文件命名、501×821 网格、3h 步长、08Z/20Z 起报、24h/48h/168h 产品窗口。最终成品应把这块做成一个稳定的数据接入服务，而不是只在用户点击时临时读文件。

最终应具备：

```text
1. 自动扫描 08Z / 20Z 新数据
2. 自动识别可用时效
3. 自动生成可编辑的 product_window
4. 自动标记缺失窗口、异常窗口
5. 自动生成 qpf_before、ptype_before
6. 自动生成 negative_qpf_mask、missing_mask
7. 形成数据接入日志
```

最终界面里，用户不应该自己判断某个窗口有没有数据，而是看到：

```text
+000~+024h  可编辑
+024~+048h  可编辑
+048~+072h  缺少 ptype
+000~+168h  可编辑，但存在 12 个负值格点
```

建议给 `product_window` 增加状态字段：

```sql
status TEXT
qc_status TEXT
negative_count INTEGER
missing_count INTEGER
data_ready_at TEXT
```

状态可以是：

```text
pending       等待数据
available     可编辑
partial        部分数据缺失
invalid        数据异常，不可编辑
archived       已归档
```

这是最终成品必备能力。

## 二、主编辑工作台

当前设计的多边形、绘制笔、笔刷、降水定值/增减/乘系数/清零、相态改雨/雪/雨夹雪/无降水，已经足够作为主编辑工具。

最终成品要加强的是**操作体验**，不是盲目增加工具。

建议最终工作台包括：

```text
1. 产品窗口选择
2. 原始图 / 当前订正图 / 差异图 / 修改区域图切换
3. 多边形 / 绘制笔 / 笔刷
4. 降水调整
5. 相态调整
6. 弱降水筛查
7. 预览 / 应用
8. 撤销 / 重做
9. 保存草稿
10. 提交审核
```

最终版本可以考虑加入 GFIE 里比较实用但不复杂的两个功能：

```text
区域移动
区域复制
```

GFIE 手册中区域移动和区域复制用于把指定区域内的数据移动或复制到目标区域，这对落区偏移订正是有价值的；但不建议一开始恢复 GFIE 的完整落区增加、落区格点化、趋势调整、要素协同等重功能。

最终工作台还应增加一个“当前格点诊断卡片”：

```text
经纬度
行列号
qpf_before
qpf_after
ptype_before
ptype_after
是否人工修改
最近一次操作 ID
```

这样值班人员能随时确认自己改的是哪个格点、改前改后是什么。

## 三、版本、草稿、审核与发布

现在你已经有 `edit_session`、`edit_operation`、`edit_version`，这是很好的基础。最终成品需要把它扩展成完整的业务状态机。

建议最终版本状态设计为：

```text
original       原始版本
draft          草稿版本
submitted      已提交审核
approved       已审核通过
rejected       审核退回
released       已正式发布
superseded     已被新版本替代
archived       已归档
```

建议流程：

```text
预报员开始编辑
→ 生成 edit_session
→ 多次 preview/apply
→ 保存草稿
→ 提交审核
→ 审核员查看订正前后图、差异图、操作日志、复盘图
→ 审核通过
→ 生成 released 版本
→ 对外或内部产品目录发布
```

最终成品里，“保存版本”和“发布版本”应该分开。

```text
保存版本：说明这次编辑结果被系统记录了
发布版本：说明这次结果可以作为正式业务产品使用
```

建议增加两张表：

```sql
CREATE TABLE review_approval (
    approval_id TEXT PRIMARY KEY,
    version_id TEXT NOT NULL,
    reviewer_id TEXT NOT NULL,
    status TEXT NOT NULL,
    comment TEXT,
    reviewed_at TEXT NOT NULL
);

CREATE TABLE release_product (
    release_id TEXT PRIMARY KEY,
    version_id TEXT NOT NULL,
    window_id TEXT NOT NULL,
    release_status TEXT NOT NULL,
    product_path TEXT,
    released_by TEXT,
    released_at TEXT
);
```

如果你们内部业务暂时不需要正式审核流，也可以做轻量版：

```text
保存草稿
保存正式版
```

但最终成品最好至少预留审核状态。

## 四、复盘绘图与复盘包

当前方案已经把复盘系统设计得比较完整：`review_payload`、`missing_fields`、`plot_task`、IFS 多要素字段、前后 24h、绘图模板、复盘数据包目录都有了。最终成品要加强的是**复盘产品管理**。

最终复盘不应只是“保存时顺便生成一张图”，而应该有一个独立的“复盘中心”。

复盘中心应支持：

```text
1. 查看某个 case 的所有窗口
2. 查看某个 window 的所有版本
3. 查看 before / after / delta / change / mask
4. 查看 IFS 复盘组合图
5. 查看缺失变量和绘图失败日志
6. 重新生成复盘图
7. 导出复盘包
8. 批量生成历史个例复盘图
```

最终复盘图建议至少有 5 类：

```text
1. 降水相态订正对比图
   before / after / delta / change

2. 当前时段 + 前后24h 过程图
   prev24 / current / next24

3. 高空环流诊断图
   500/700/850 高度、风、温湿组合

4. 修改行为复盘图
   edit_mask + 操作统计 + 相态变化矩阵

5. 综合复盘大图
   业务领导或复盘会直接看的一张总图
```

当前设计里“复盘图模板管理”放在后期，我认为作为最终成品它应当进入正式范围。否则每次换图型都要改代码。

建议模板配置最终做到：

```json
{
  "template_id": "snow_phase_review_v1",
  "template_name": "雨雪相态订正复盘图",
  "required_fields": [
    "qpf_before",
    "ptype_before",
    "qpf_after",
    "ptype_after",
    "delta_qpf",
    "change_ptype",
    "z500",
    "t850",
    "rh700",
    "u850",
    "v850"
  ],
  "optional_fields": [
    "mslp",
    "t2m",
    "tcwv"
  ],
  "allow_partial_success": true
}
```

## 五、质量控制与一致性检查

当前方案里的一致性检查主要是：

```text
qpf <= threshold 时，ptype 必须为无降水
ptype = 无降水但 qpf > threshold 时提示
qpf_after 小于 0 自动修正
```

作为最终成品，这还不够。最终至少要有三层质控。

### 第一层：硬规则

必须自动执行：

```text
1. qpf < 0 → 置为 0 或标记异常
2. qpf <= 0.1 → ptype = 0
3. ptype = 0 且 qpf > 0.1 → 保存前提示必须处理
4. ptype 不在 0/1/2/3 → 标记非法
5. 缺测格点不可编辑
6. negative_qpf_mask 不参与编辑和统计
```

### 第二层：业务提示

不强制，但保存前提醒：

```text
1. 单次操作影响面积过大
2. 降水总量变化过大
3. 大面积雨直接改雪
4. 大量无降水格点被改成有降水
5. 当前版本与上一正式版本差异过大
```

### 第三层：复盘质控

用于后续分析：

```text
1. 哪些区域经常被人工修改
2. 哪些窗口经常修改量级
3. 哪些窗口经常雨改雪
4. 人工修改是否集中在某些天气型
5. 复盘图是否存在缺失变量
6. 绘图任务是否经常 partial_success
```

最终成品中，这些质控结果应当进入 `edit_summary.csv` 或数据库，而不是只在前端弹窗里显示。

## 六、操作日志与审计

当前操作日志已经有 `session_id`、`sequence_no`、工具、变量、操作、参数、影响格点数、统计等。最终成品要把它升级成“可审计、可查询”的操作历史。

最终应能回答：

```text
某个版本是谁改的？
什么时候开始改？
一共做了几步？
每一步改了哪里？
改前改后分别是什么？
撤销了哪些操作？
最后哪几步生效？
这个版本是否经过审核？
是否正式发布？
复盘图是否生成成功？
```

建议操作日志最终保存两种 mask：

```text
mask_geometry_json：用户画的几何形状
mask_raster_path：实际影响格点的布尔数组
```

这是最终成品很重要的一点。因为以后复盘时你不仅要知道“哪些格点变了”，还要知道“预报员是怎么画出来的”。

## 七、历史样本库与分析看板

这是最终成品和普通编辑器最大的差别。

你这个系统的价值不只是当班修改图，而是**长期积累人工订正样本**。最终成品应有一个“历史分析”模块。

可以分析：

```text
1. 不同起报时次的修改频率
2. 24h / 48h / 168h 哪类产品修改最多
3. 哪些区域经常被改
4. 哪些相态转换最常见
5. 雨改雪、雪改雨、雨改雨夹雪的空间分布
6. 人工订正对降水总量的平均影响
7. 哪些 IFS 形势下容易发生大幅订正
8. 复盘图缺失变量统计
9. 预报员操作行为统计
```

这部分不用一开始做得很复杂，但最终成品至少要能导出：

```text
case_id
window_id
version_id
init_time
accum_hours
start_lead
end_lead
modified_grid_count
modified_area
qpf_sum_before
qpf_sum_after
ptype_transition_matrix
operation_count
review_plot_status
```

如果后续要做自动订正模型，这些就是训练样本的基础。

建议最终界面分成 5 个一级菜单：

```text
1. 网格编辑
2. 版本审核
3. 复盘中心
4. 历史分析
5. 系统管理
```

```text

编辑

查看待审核版本

降水总量订正统计
绘图任务成功率
复盘样本导出
```

## 系统管理

```text
用户角色
路径配置
色标配置
绘图模板
数据状态
磁盘空间
任务日志
```

这样才像最终成品，而不是只像一个编辑页面。

# 7. 最终成品是否要加入 GFIE 那些功能？

我的建议是：**选择性加入，不要全量加入。**

## 可以加入

```text
1. 区域锁定
2. 区域移动
3. 区域复制
4. 圆形工具
5. 格点等值线显示，不一定编辑
6. 上传/分发思想，转成正式产品发布
```

GFIE 的区域锁定可以限制只能修改指定区域，能减少误操作；区域移动/复制对落区偏移纠正有帮助；上传分发思想可以转化为正式产品发布。

## 不建议加入

```text
1. 背景数据选择
2. 多要素协同编辑
3. 格点趋势调整
4. 站点修改
5. 短临预警制作
6. 预警信号制作
7. 天气现象数字编码式编辑
```

这些会把系统重新推向 GFIE 那种综合平台，和你现在“累计降水 + 相态编辑 + IFS 复盘”的目标不一致。

# 8. 最终成品验收标准应提高

当前验收标准偏功能验收。最终成品还应增加运行验收。

建议最终验收分 6 类。

## 功能验收

```text
能加载窗口
能编辑
能保存
能撤销重做
能出图
能复盘
能审核发布
能导出复盘包
```

## 数据验收

```text
qpf 差分正确
ptype 规则一致
before/after/delta/mask 完整
operation_log 完整
manifest 完整
原始数据不可覆盖
```

## 复盘验收

```text
IFS 变量缺失时可 partial_success
复盘图可重新生成
plot_log 可追踪
复盘包可独立打开
```

## 权限验收

```text
预报员不能删除正式版本
审核员可以审核发布
管理员可以管理配置
所有操作可追踪
```

## 性能验收

```text
窗口加载 ≤ 5 秒
普通多边形预览 ≤ 3 秒
笔刷响应 ≤ 1 秒
保存版本 ≤ 10 秒，不含复杂复盘图
复盘绘图有任务进度
```

## 运维验收

```text
服务重启后 session/版本不丢
绘图失败可重试
归档目录可清理
配置变更有记录
磁盘不足有告警
```

# 9. 我的最终建议

你现在的方案可以看作：

```text
最终成品的核心版设计
```

而不是简单 MVP。

如果要把它完善成最终成品，重点不是继续加很多编辑工具，而是补齐：

```text
1. 自动数据接入与 QC
2. ptype 累计窗口生成规则
3. 审核发布流程
4. 复盘中心
5. 历史分析看板
6. 绘图任务队列与失败重试
7. 权限与审计
8. 配置版本管理
9. 归档包索引与批量导出
10. 运维监控与备份恢复
```

最终产品应该是：

> **一个固定 0.05° 网格的累计降水与雨雪相态业务订正平台，主编辑只处理 qpf 和 ptype，复盘系统接入 EC IFS 多要素诊断场，完整支持数据接入、人工订正、版本审核、产品发布、复盘归档、历史分析和运维监控。**

这样做，既不会变成 GFIE 那种过重的综合平台，也能达到真正“最终成品”的业务化水平。
