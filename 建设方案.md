# 固定区域 5km 累计降水与雨雪相态网格编辑及复盘系统建设方案

**版本**：V1.1  
**日期**：2026-05-15  
**适用范围**：固定区域 5km 网格、24h/48h/168h 累计降水与相态产品订正、EC IFS 多要素复盘绘图  
**方案目标**：建设一个轻量、可落地、可追溯、可复盘的网格编辑系统。系统只对累计降水和相态进行交互订正；EC IFS 的高空高度场、风、温度、湿度及其他变量不进入主编辑流程，而作为复盘绘图和后续分析的数据输入。  
**技术选型**：前端 Vue 3 + OpenLayers + TDesign，后端 Python FastAPI，部署 Docker  
**投影坐标系**：EPSG:4326（等经纬度投影）

---

## 1. 项目背景

当前已有基础条件包括：

1. 固定区域底图，包括 PNG 底图以及绘制该底图的代码。

3. 5km 分辨率 DEM 数据（待提供）。

5. 已有复盘绘图代码（Flask + Matplotlib + Cartopy），包括环流形势、降水、要素场、地面场、剖面图等绘图脚本，运行在 192.168.2.80:8080。
6. 已有 WebGIS 前端参考项目（Vue 3 + OpenLayers + TDesign），包含网格图层渲染、站点图层、WebGL 瓦片、投影切换、时间轴等功能。
7. 已有后端参考项目（Java Spring Boot），包含 GRIB 解析、数据入库、站点管理等功能。

经对样本数据（data/量级、data/phase、data/EC预报）核查，确认如下事实：

| 核查项 | 建设方案 V1.0 原值 | 实际值 | 处理 |

| 网格尺寸 | nx=420, ny=360 | **501 行 × 821 列** | 以实际数据为准 |

| DEM 数据 | 已有 | **待提供** | 等待 |

本项目不建设综合性气象网格平台，也不建设多模式、多要素交互浏览系统。建设重点是：

> 在固定区域 5km 网格上，对 24h、48h、168h 可控累计时段的降水量和相态结果进行人工订正，并把订正前后格点场、差异场、修改掩膜、操作日志、基础出图和 IFS 多要素复盘组合图完整保存下来。

## 2. 对陕西 GFIE 手册的吸收与取舍

陕西省格点预报智能编辑系统 GFIE 的用户手册中，系统具有编辑工具栏、图例栏、背景数据选择、时间选择、模块菜单、要素菜单、通用工具栏、图层属性菜单栏、状态栏等组成部分。其编辑工具包括主客观融合、落区增加、权重修改、多边形、绘制笔、圆形、笔刷、区域移动、区域复制、筛查、要素协同等。

本项目不照搬 GFIE 的完整平台形态，而是吸收其中适合本项目边界的编辑思想。

### 2.1 建议吸收的 GFIE 功能思想

| GFIE 功能思想 | 在本系统中的转化 |
|---|---|
| 时间选择 | 转化为 24h/48h/168h 累计长度与起止时效选择 |
| 多边形 | 用于大范围圈选并修改累计降水或相态 |
| 绘制笔 | 用于沿雨雪边界、降水边界进行不规则区域修正 |
| 笔刷 | 用于小范围修补量级、相态、孤立斑块 |
| 筛查 | 转化为弱降水清零、选区内相态批处理、选区内量级统一 |
| 撤销、恢复、保存 | 转化为操作级撤销、重做、版本保存和复盘归档 |
| 状态栏显示格点值 | 转化为当前鼠标格点诊断栏，只显示 qpf、ptype、修改状态等核心信息 |
| 区域移动/复制 | 作为二期功能，用于落区整体偏移或复制 |

### 2.2 不建议纳入首版的 GFIE 功能

| GFIE 功能 | 不纳入原因 |
|---|---|
| 背景数据选择 | 本项目没有多背景场选择需求，输入产品固定 |
| 多要素交互显示 | IFS 多要素只用于复盘绘图，不进入主编辑界面 |
| 格点趋势调整 | 本项目只编辑累计结果，不拆分到 3h/6h 分段，不做时间趋势回推 |
| 要素协同 | 首版只编辑降水和相态，不做多要素协同修改 |
| 站点修改 | 本项目编辑对象是格点产品，不是站点产品 |
| 预警信号制作 | 不属于本项目建设目标 |
| 短临预警制作 | 不属于本项目建设目标 |
| 大量独立工具按钮 | 容易造成操作复杂，建议改为任务式工具面板 |

## 3. 建设目标

### 3.1 总体目标

建设一个固定区域、固定 5km 网格的累计降水与雨雪相态网格编辑系统。系统支持用户选择累计时段，对降水量级和相态进行区域化人工订正，自动生成订正前后对比、差异图、相态变化图、修改掩膜、操作日志及 IFS 多要素复盘组合图。

### 3.2 核心目标

1. **可编辑**：支持对累计降水量和相态进行多边形、绘制笔、笔刷等交互式订正。
2. **可控时段**：支持 24h、48h、168h 累计长度，并允许控制起始时效和结束时效。
3. **可出图**：复用已有底图和绘图代码，生成订正前、订正后、差异、相态变化、复盘组合图。
4. **可追溯**：保存每一步操作，包括操作人、时间、工具、范围、参数、影响格点数、订正前后统计。
5. **可复盘**：保存订正前后格点数据、IFS 多要素诊断场切片、绘图输入清单、绘图配置和复盘结果图。
6. **可扩展**：后续可接入更多 IFS 变量、更多复盘模板、实况检验和批量个例分析。

## 6. 业务流程

### 6.1 编辑流程

```text
1. 用户选择起报时间
2. 用户选择累计长度：24h / 48h / 168h
3. 用户选择起始时效
4. 系统自动计算结束时效
5. 系统加载 qpf_before、ptype_before
6. 系统初始化 qpf_after = qpf_before，ptype_after = ptype_before
7. 用户选择多边形、绘制笔或笔刷
8. 用户选择编辑动作：降水定值、增加、减少、乘系数、清零，或相态改雨/雨夹雪/雪/无降水
9. 系统生成预览
10. 用户确认应用
11. 系统记录操作日志
12. 用户保存版本
13. 系统生成差异场、相态变化、修改掩膜
14. 系统调用基础绘图代码
15. 系统调用复盘绘图代码
16. 系统保存复盘数据包
```

### 6.2 复盘流程

```text
1. 接收已保存的 edit_version
2. 定位当前累计时段
3. 计算前 24h 与后 24h 窗口
4. 读取当前窗口 qpf/ptype 订正前后数据
5. 读取前后 24h 的 qpf/ptype 数据
6. 读取对应 IFS 复盘变量
7. 生成 plot_input_manifest.json
8. 调用复盘绘图代码
9. 保存 review_composite.png 及其他复盘图
10. 记录复盘产品信息
```

## 7. 产品窗口设计

### 7.1 累计时段控制

系统只支持三种累计长度：

```text
24h
48h
168h
```

累计时段由两个参数控制：

```text
accum_hours：累计长度，取值 24 / 48 / 168
start_lead：起始预报时效
end_lead：结束预报时效，end_lead = start_lead + accum_hours
```

示例：

| 累计长度 | 起始时效 | 结束时效 | 窗口含义 |
|---:|---:|---:|---|
| 24h | +000h | +024h | 起报后 0 至 24 小时累计降水 |
| 24h | +024h | +048h | 起报后 24 至 48 小时累计降水 |
| 48h | +000h | +048h | 起报后 0 至 48 小时累计降水 |
| 48h | +048h | +096h | 起报后 48 至 96 小时累计降水 |
| 168h | +000h | +168h | 起报后 0 至 168 小时累计降水 |

### 7.2 窗口 ID 命名

建议使用统一命名规则：

```text
window_id = {init_time}_ACC{accum_hours}_{start_lead}_{end_lead}
```

示例：

```text
2026010108_ACC24_024_048
2026010108_ACC48_000_048
2026010108_ACC168_000_168
```

## 9. 相态编码（已确认）

经分析 `reference/cdsz` 中 `GribToPng.pngCombinePtype()` 方法（GribToPng.java:250-261），**已有系统的相态编码**如下：

### 9.1 编码定义

| 编码 | 含义 | 渲染逻辑 | 数据来源 |
|---:|---|---|---|
| 0 | 无降水 | 透明（降水量 <0.1 时 alpha=0） | ptype_999_revised 文件 |
| 1 | 雨 | `getRainRgbaValue()` 绿蓝紫分级 | ptype_999_revised 文件 |
| 2 | 雪 | `getSnowRgbaValue()` 灰色分级 | ptype_999_revised 文件 |
| 3 | 雨夹雪 | `getRainSnowRgbaValue()` 固定粉紫色 | ptype_999_revised 文件 |

> **注意**：此编码与 cdsz 中 `ptypeMap`（站点天气现象：0=晴,1=雨,2=雪,3=少云,4=多云,5=阴）**不是同一套体系**。`ptypeMap` 是站点预报天气现象，包含云量类别；`ptype_999_revised` 是纯降水相态编码，仅涉及降水类型。

> **实际数据现状**：当前样本数据（5 月中旬）出现 0、1、2 三种值。编码 3（雨夹雪）在冬季数据中预期出现。

### 9.2 色标配置（已有代码定义）

#### 雨（ptype=1）：`getRainRgbaValue()`

| 量级(mm) | RGB | 颜色描述 |
|---|---|---|
| <0.1 | 透明 | — |
| 0.1 - 10 | (170, 240, 144) | 浅绿 |
| 10 - 25 | (61, 172, 5) | 深绿 |
| 25 - 50 | (104, 185, 249) | 浅蓝 |
| 50 - 100 | (5, 4, 247) | 深蓝 |
| 100 - 250 | (253, 4, 255) | 紫红 |
| ≥250 | (253, 4, 255) | 紫红 |

#### 雪（ptype=2）：`getSnowRgbaValue()`

| 量级(mm) | RGB | 颜色描述 |
|---|---|---|
| <0.1 | 透明 | — |

## 11. 编辑功能详细设计

### 11.1 多边形编辑

适用场景：

```text
大范围降水量级增强或减弱
大范围相态从雨改雪
大范围清零弱降水
```

交互流程：

```text
1. 点击多边形工具
2. 左键添加顶点
3. 双击完成绘制
4. 右侧选择操作类型和参数
5. 点击预览
6. 确认后点击应用
```

不建议使用“画完即变”的模式，所有操作均需先预览再应用。

### 11.2 绘制笔编辑

适用场景：

```text
沿雨雪边界修正相态
沿山脉或降水边界画不规则狭长区域
修补不规则降水落区
```

绘制笔生成的是一个带宽度的线状缓冲区，缓冲区内格点进入 mask。

### 11.3 笔刷编辑

适用场景：

```text
擦除孤立雪点
补充局地雪区
局地增加或减少降水量
小范围相态修正
```

笔刷参数：

```text
半径：1 / 2 / 3 / 5 个格点
模式：硬边界 / 软边界，首版可只做硬边界
动作：降水增加/减少/定值/清零，或相态设置
快捷操作：Ctrl + 鼠标滚轮调整笔刷大小
```

### 11.4 降水编辑算法

```python
# mask 为当前选区布尔数组

# 定值
qpf_after[mask] = value

# 增加
qpf_after[mask] = qpf_after[mask] + delta

# 减少
qpf_after[mask] = np.maximum(qpf_after[mask] - delta, 0)

# 乘系数
qpf_after[mask] = qpf_after[mask] * factor

# 清零
qpf_after[mask] = 0
ptype_after[mask] = 0
```

### 11.5 相态编辑算法

```python
# qpf_threshold 默认 0.1mm（与 GribToPng 中 value < 0.1 透明处理一致），可配置
valid_precip = qpf_after > qpf_threshold

# target_ptype: 0=无降水, 1=雨, 2=雪, 3=雨夹雪
ptype_after[mask & valid_precip] = target_ptype

# 无降水区域强制置为 0
ptype_after[~valid_precip] = 0
```

### 11.6 筛查清零

```python
screen_mask = selected_mask & (qpf_after <= threshold)
qpf_after[screen_mask] = 0
ptype_after[screen_mask] = 0
```

筛查阈值默认：

```text
0.1 mm
```

可根据业务配置为：

```text
0.1 mm / 0.5 mm / 1.0 mm / 自定义
```

## 12. 一致性规则

首版只做最基础、必要的一致性规则。

### 12.1 硬规则

| 规则 | 处理方式 |
|---|---|
| qpf <= threshold 时，ptype 必须为无降水 | 自动修正 |
| ptype = 无降水但 qpf > threshold | 提示用户，允许选择保留降水或清零 |
| qpf_after 小于 0 | 自动修正为 0 |

### 12.2 软提示

| 规则 | 处理方式 |
|---|---|
| 大面积相态被修改 | 保存时提示影响格点数和面积 |
| 降水总量变化过大 | 保存时提示订正前后总量差异 |
| 相态变化跨类别过大，例如雨(1)直接改雪(2)跳过雨夹雪(3) | 记录到操作日志，不阻止保存 |

不引入温度、湿度、地形等一致性校验，避免系统超出当前数据编辑边界。

## 14. 复盘系统设计

### 14.1 复盘系统定位

复盘系统不参与人工编辑，只负责把编辑结果和 IFS 多要素诊断场组织成绘图输入，并调用用户提供的绘图代码生成组合图。

### 14.2 当前时段与前后 24h 定义

当前编辑窗口：

```text
current_window = [start_lead, end_lead]
```

前 24h：

```text
prev24_window = [start_lead - 24, start_lead]
```

后 24h：

```text
next24_window = [end_lead, end_lead + 24]
```

示例：

```text
当前窗口：+024h ~ +048h，24h累计
前24h：+000h ~ +024h
后24h：+048h ~ +072h
```

如果前后 24h 数据不存在，则复盘图对应区域标注“无数据”，并在 manifest 中记录缺失。

### 14.3 IFS 诊断场时间取样

累计降水是一个时间段，而高度场、风场、温度场、湿度场通常是某个有效时刻的预报场。因此需要制定时间取样策略。

建议预留配置：

```json
{
  "review_time_policy": {
    "current_window": ["start", "middle", "end"],
    "prev24_window": ["end"],
    "next24_window": ["start"],
    "default_pressure_field_time": "middle"
  }
}
```

首版可简化为：

```text
当前窗口：取中间时刻和结束时刻
前24h：取结束时刻
后24h：取开始时刻
```

具体取样规则可在接入绘图代码后再精确调整。

## 24. 风险与应对

| 风险 | 表现 | 应对 |
|---|---|---|
| 网格与底图映射不准 | 鼠标选区落不到正确格点 | 建立 grid_definition.json，并做格点拾取测试 |
| 只保存图片不保存格点 | 后续无法复盘和统计 | 强制保存 qpf/ptype before/after 和 delta/mask |
| 累计窗口命名混乱 | 24h/48h/168h 数据混淆 | 统一 window_id 命名规则 |
| qpf 与 ptype 不一致 | 0 降水仍显示雪 | 设置 qpf_threshold 并自动清理相态 |
| 绘图代码输入不统一 | 复盘图生成失败 | 建立 plot_input_manifest.json 作为适配层 |
| IFS 变量过多导致存储膨胀 | 复盘包过大 | 只保存本次绘图用到的变量切片，同时记录原始路径 |
| 后续测试数据格式不确定 | 接入延期 | 先做数据适配器接口，等测试数据到位后落地 |
| 操作误修改 | 难以回退 | 每次操作保存历史，支持撤销/重做 |
| 版本覆盖 | 原始数据丢失 | 原始版本不可变，保存版本只新增不覆盖 |

## 28. 最终结论

本系统应建设为一个轻量但数据留痕严格的网格编辑与复盘系统。核心原则是：

> 主编辑界面只处理累计降水和相态；IFS 多要素诊断场不进入编辑流程，只进入复盘绘图和归档流程。

首版最重要的闭环是：

```text
选择累计窗口
→ 显示累计降水和相态
→ 区域化修改量级或相态
→ 预览和应用
→ 保存订正前后格点
→ 生成差异和修改掩膜
→ 记录操作日志
→ 调用绘图代码
→ 生成并归档复盘数据包
```

这样既能满足当前业务需要，又不会把系统做成过重的综合气象平台。后续在你提供具体绘图代码和测试数据后，重点工作应放在：

```text
1. 明确 IFS 复盘变量清单
2. 明确变量文件格式和时效规则
3. 定义 plot_input_manifest.json
4. 封装绘图代码适配器
5. 打通保存版本后自动生成复盘图的流程
```

## 附录 D：Docker 部署方案

### D.1 开发流程

```text
本地开发 → 本地测试 → 打镜像 → 部署到生产内网
```

开发阶段前后端分别独立运行：

```text
前端：npm run dev（Vite 开发服务器，热更新）
后端：uvicorn main:app --reload（FastAPI 开发模式）
```

### D.2 生产部署结构

```text
docker-compose.yml
├── frontend（Nginx 容器）
│   ├── Vue 构建产物
│   └── nginx.conf（反向代理 /api → backend）
├── backend（FastAPI 容器）
│   ├── Python 3.11 + NumPy/xarray/Matplotlib/Cartopy
│   ├── 绘图字体（simhei.ttf）
│   └── Cartopy 地图数据（离线）
└── volumes
    ├── data/（格点数据挂载）
    ├── archive/（归档目录挂载）
    └── db/（SQLite 数据库文件挂载）
```

### D.3 关键注意事项

1. Matplotlib + Cartopy 镜像较大（约 1-2 GB），建议使用多阶段构建减小体积。
2. 生产内网无外网访问，Cartopy 地图数据和字体必须离线打入镜像。
3. 格点数据和归档目录通过 Docker Volume 挂载，不打入镜像。
4. 前端构建产物为静态文件，由 Nginx 容器直接托管，同时反向代理 API 请求到后端容器。
