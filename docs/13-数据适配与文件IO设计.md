# 13. 数据适配与文件 IO 设计

## 13.1 设计目标

本文件定义原始数据读取、累计窗口构建、相态合成、QC mask、缓存、归档路径和文件格式，保证数据接入、编辑、复盘、发布使用同一套文件口径。

## 13.2 数据源

| 数据 | 文件模式 | 用途 |
|---|---|---|
| tp | `tp_999_deeplearning_YYYYMMDDHH_hhh.txt` | 从起报起累计降水，差分得到 qpf_window |
| pr | `pr_999_deeplearning_YYYYMMDDHH_hhh.txt` | 降水概率，暂不编辑，可保留为复盘或后续辅助 |
| ptype | `ptype_999_revised_YYYYMMDDHH_hhh.txt` | 3h 相态，合成累计窗口 ptype_before |
| IFS | GRIB/GDS/其他 | 复盘绘图变量，不进入编辑主流程 |

固定网格：

```text
ny = 501
nx = 821
lon_min = 70
lon_max = 111
lat_min = 25
lat_max = 50
dlon = 0.05
dlat = 0.05
projection = EPSG:4326
row_col_order = y_desc_x_asc
```

## 13.4 qpf_window 构建

函数：

```python
def build_qpf_window(case_id: str, start_lead: int, end_lead: int, config: ProductConfig) -> QpfBuildResult:
    ...
```

规则：

```text
qpf_before = tp(end_lead) - tp(start_lead)
```

`start_lead=000` 优先级：

```text
1. 如果 tp_000 文件存在，读取实际文件；
2. 如果不存在且 allow_zero_start_lead_fallback=true，使用全零数组；
3. 否则窗口 invalid。
```

返回：

```python
@dataclass
class QpfBuildResult:
    qpf: np.ndarray
    start_tp_path: Path | None
    end_tp_path: Path
    negative_mask: np.ndarray
    missing_mask: np.ndarray
    negative_count: int
    negative_min_value: float | None
    negative_abs_max: float | None
    missing_count: int
    qc_status: Literal['pass', 'warn', 'fail']
```

负值处理：

```text
- 负值格点不参与编辑和统计。
- qpf_before 原始值可保留在 raw_qpf_before，用于数据诊断。
- 渲染和编辑使用 masked qpf，其中负值格点透明或斜线标注。
```

建议保存：

```text
original/qpf_before.npz
original/qpf_raw_before.npz，可选
original/negative_qpf_mask.npz
original/missing_mask.npz
original/qpf_build_manifest.json
```

`qpf_build_manifest.json` 记录构建溯源：

```json
{
  "window_id": "2026010108_ACC24_024_048",
  "start_lead": 24,
  "end_lead": 48,
  "start_tp_path": "tp_999_YYYYMMDDHH_024.txt",
  "end_tp_path": "tp_999_YYYYMMDDHH_048.txt",
  "start_tp_sha256": "a1b2c3...",
  "end_tp_sha256": "d4e5f6...",
  "start_lead_fallback": false,
  "negative_count": 0,
  "negative_min_value": null,
  "missing_count": 0,
  "qc_status": "pass",
  "built_at": "2026-01-01T08:15:00Z"
```

## 13.5 ptype_window 构建

函数：

```python
def build_ptype_window(case_id: str, start_lead: int, end_lead: int, config: ProductConfig) -> PtypeBuildResult:
    ...
```

时效范围：

```text
(start_lead, end_lead]
```

示例：

```text
024-048h → 027,030,033,036,039,042,045,048
```

步骤：

1. 读取每个 3h ptype 文件；缺失则记录到 `ptype_missing_leads`。
2. 对每个 3h 时效读取 tp[t] 和 tp[t-3]，计算 qpf_step：

```text
qpf_step[t] = tp[t] - tp[t-3]
```

3. 中间 tp 的 QC（逐时效、逐格点）：

| 情况 | 处理 |
|---|---|
| tp[t] 或 tp[t-3] 文件缺失 | 该 lead 整体不参与 ptype 合成，记录到 `tp_missing_leads` |
| tp[t] 或 tp[t-3] 格点为 NaN | 该 lead 该格点不参与相态统计 |
| qpf_step[t] < 0（逐格点） | 该 lead 该格点不参与相态统计，计入 `step_negative_count` |

4. 仅满足以下条件的 (lead, grid_point) 参与相态统计：
   - tp[t] 和 tp[t-3] 均存在且非 NaN
   - qpf_step[t] >= 0
   - qpf_step[t] > qpf_threshold

5. 逐格点设置：

```text
ptype=1 → has_rain
ptype=2 → has_snow
ptype=3 → has_rain + has_snow
```

6. 合成：

| has_rain | has_snow | ptype_before |
|---:|---:|---:|
| 0 | 0 | 0 |
| 1 | 0 | 1 |
| 0 | 1 | 2 |
| 1 | 1 | 3 |

返回：

```python
@dataclass
class PtypeBuildResult:
    ptype: np.ndarray
    valid_leads: list[int]
    ptype_missing_leads: list[int]
    tp_missing_leads: list[int]
    step_negative_count: int
    step_nan_count: int
    effective_lead_count: int
    qc_status: Literal['pass', 'warn', 'fail']
```

`effective_lead_count` = 窗口内 lead 总数 - ptype 缺失 - tp 缺失（同一 lead 两种缺失只计一次）。

状态判定：

```text
effective_lead_count == 0 → product_window invalid，qc_status=fail
effective_lead_count < total_leads → product_window partial，qc_status=warn
step_negative_count > 0 → qc_status 至少 warn
无缺失且无负值 → available/pass
```

构建完成后保存 `original/ptype_source_manifest.json`：

```json
{
  "window_id": "2026010108_ACC24_024_048",
  "start_lead": 24,
  "end_lead": 48,
  "all_leads": [27, 30, 33, 36, 39, 42, 45, 48],
  "used_leads": [27, 30, 33, 36, 39, 42, 45, 48],
  "ptype_missing_leads": [],
  "tp_missing_leads": [],
  "effective_lead_count": 8,
  "step_negative_count": 0,
  "step_nan_count": 0,
  "qpf_threshold": 0.1,
  "synthesis_rule": "has_rain_has_snow",
  "qc_status": "pass",
  "built_at": "2026-01-01T08:15:00Z"
}
```

产物分类：

| 类别 | 用途 | 格式 |
|---|---|---|
| fields/*.npz | 内部系统存储和回看 | numpy npz |
| fields/*.txt | 面向下游业务系统（与原始 tp/ptype 文件格式一致：501 行 × 821 列 CSV） | 纯文本 CSV |
| derived/ | 审计和复盘分析 | npz + csv |
| images/ | 人工查看和归档 | PNG |
| review/ | 复盘组合图 | PNG |

txt 输出规则：

```text
qpf_after.txt：501×821 CSV，逗号分隔，Float 保留 2 位小数，单位 mm
ptype_after.txt：501×821 CSV，逗号分隔，整数 0/1/2/3
行列顺序与原始文件一致（y_desc_x_asc），保证下游系统可直接替换使用
```

ReleaseService 必须：
1. 从 version 目录复制 npz 文件到 released 目录
2. 将 npz 转换为 txt 格式写入 fields/
3. 从 version/images 复制图片
4. 从 review 目录复制 review_composite
5. 生成 product_manifest.json
6. 全部完成后再更新 DB release_product 记录

## 13.9 PathBuilder 规则

路径必须由 `PathBuilder` 统一生成。PathBuilder 输入 `case_id/window_id/session_id/version_id/review_id`，输出 pathlib.Path。

路径生成不应依赖前端传入路径，避免路径穿越和归档混乱。

## 13.11 文件写入原子性

所有正式归档文件采用：

```text
write tmp → fsync → rename
```

避免服务中断时出现半写入文件。

伪代码：

```python
def atomic_write_npz(path: Path, **arrays):
    tmp = path.with_suffix(path.suffix + '.tmp')
    with open(tmp, 'wb') as f:
        np.savez_compressed(f, **arrays)
        f.flush()
        os.fsync(f.fileno())
    os.replace(tmp, path)
```

注意：`np.savez_compressed` 接收文件名字符串时会自动追加 `.npz` 后缀，传入文件句柄可避免此问题。
