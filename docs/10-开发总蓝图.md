# 10. 开发总蓝图

## 10.1 文档定位

本文是进入代码开发前的总控文档，用于统一领域对象、模块边界、端到端流程和跨模块追踪矩阵。业务功能范围以 `docs/01-09` 为准；工程实现口径以 `docs/10-20` 为准。

目标是保证：

```text
同一个业务对象，在数据库、API、后端服务、前端 Store、文件归档、测试用例中使用同一套命名和生命周期。
```

## 10.3 核心领域对象

## 10.5 生命周期总览

### 10.5.1 数据窗口生命周期

```text
扫描数据目录
  ↓
创建 ForecastCase
  ↓
枚举 24h / 48h / 168h 窗口
  ↓
构建 qpf_before / ptype_before
  ↓
执行 QC
  ↓
ProductWindow: pending → available / partial / invalid
```

### 10.5.2 编辑生命周期

```text
选择 ProductWindow
  ↓
POST /api/session/start
  ↓
EditSession: editing
  ↓
多次 preview / apply
  ↓
undo / redo，可选
  ↓
POST /api/version/save
  ↓
EditSession: saved
EditVersion: draft
```

### 10.5.3 审核发布生命周期

```text
EditVersion: draft
  ↓ submit
EditVersion: submitted
  ↓ approve / reject
EditVersion: approved / rejected
  ↓ release，仅 approved 可发布
EditVersion: released
ReleaseProduct: active
  ↓ 新版本发布
旧 EditVersion: superseded
旧 ReleaseProduct: superseded
```

### 10.5.4 复盘生命周期

```text
保存或发布版本
  ↓
生成 review_payload / plot_input_manifest
  ↓
提交 plot_queue
  ↓
PlotTask: pending → running → success / partial_success / failed
  ↓
生成 review_product
  ↓
复盘中心查看 / 重生成 / 导出复盘包 / 历史统计
```

## 10.8 开发原则

1. **领域对象先行**：新增功能必须先映射到已有领域对象，无法映射时再新增对象。
2. **状态机受控**：所有状态改变必须通过 Service 方法执行，不允许直接更新表字段。
3. **文件路径统一生成**：所有路径通过 `PathBuilder` 生成，禁止在业务代码中手写归档路径。
4. **preview 不入正式库**：preview 只做短期缓存；apply 后才形成 edit_operation。
5. **版本不可覆盖**：edit_version、release_product、review_product 只新增或置状态，不覆盖历史。
6. **绘图可失败，版本不可丢**：绘图失败不得影响版本保存，失败状态进入 review_product。
7. **前后端共享枚举**：状态、角色、ptype、错误码应通过配置或生成文件保持一致。
8. **审计独立于业务日志**：业务日志记录操作内容；audit_log 记录安全和关键动作。

## 10.9 与 GFIE 的关系

本系统借鉴 GFIE 的多边形、绘制笔、笔刷、筛查、撤销、保存、上传/分发思想，但不复制其多背景场、多要素协同、短临预警、预警信号等完整平台功能。GFIE 中“编辑工具 + 状态栏 + 通用工具栏”的结构可作为交互参考，本系统实现时应拆分为：mask 生成、数值编辑、操作日志和版本留痕四个工程单元。
