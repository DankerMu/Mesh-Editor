## 7. 产品窗口设计

### 7.1 累计时段控制

系统只支持三种累计长度：

```text
24h
48h
168h
```

累计时段由两个参数控制：

```text
accum_hours：累计长度，取值 24 / 48 / 168
start_lead：起始预报时效
end_lead：结束预报时效，end_lead = start_lead + accum_hours
```

示例：

| 累计长度 | 起始时效 | 结束时效 | 窗口含义 |
|---:|---:|---:|---|
| 24h | +000h | +024h | 起报后 0 至 24 小时累计降水 |
| 24h | +024h | +048h | 起报后 24 至 48 小时累计降水 |
| 48h | +000h | +048h | 起报后 0 至 48 小时累计降水 |
| 48h | +048h | +096h | 起报后 48 至 96 小时累计降水 |
| 168h | +000h | +168h | 起报后 0 至 168 小时累计降水 |

### 7.2 窗口状态机

每个产品窗口由数据接入服务自动创建和维护状态。

状态定义：

| 状态 | 含义 | 可进入编辑 |
|---|---|:---:|
| `pending` | 等待数据，tp/ptype 文件尚未齐全 | 否 |
| `available` | 数据齐全，QC 通过，可编辑 | 是 |
| `partial` | 部分数据缺失（如某些 3h ptype 文件），可编辑但有告警 | 是 |
| `invalid` | 数据异常不可修复（如 end_lead 文件缺失），不可编辑 | 否 |
| `archived` | 已归档，只读 | 否 |

状态转换：

```text
[数据扫描触发]
    ↓
pending ──(end_lead tp 到达)──→ 检查所有依赖文件
    ├── 全部齐全 + QC 通过 → available
    ├── 部分 ptype 缺失    → partial
    └── end_lead tp 缺失   → invalid

available / partial ──(用户保存版本)──→ 状态不变
available / partial ──(归档操作)──→ archived
```

QC 自动标记：

| 检查项 | 触发条件 | 标记字段 |
|---|---|---|
| 差分负值 | `qpf_before` 存在负值格点 | `negative_count`、`negative_min_value`、`negative_abs_max` |
| 缺测格点 | tp/ptype 文件中存在 NaN | `missing_count` |
| ptype 文件缺失 | 窗口内某些 3h ptype 文件不存在 | `ptype_missing_leads`（JSON 数组） |

### 7.3 窗口 ID 命名

使用统一命名规则：

```text
window_id = {init_time}_ACC{accum_hours}_{start_lead}_{end_lead}
```

示例：

```text
2026010108_ACC24_024_048
2026010108_ACC48_000_048
2026010108_ACC168_000_168
```

## 8. 数据对象设计

### 8.1 原始数据文件命名（已确认）

| 文件类型 | 命名格式 | 说明 |
|---|---|---|
| 累积降水量 | `tp_999_deeplearning_YYYYMMDDHH_hhh.txt` | 累积值，需差分得时段降水 |
| 降水概率 | `pr_999_deeplearning_YYYYMMDDHH_hhh.txt` | 0.0-1.0，辅助参考，不进入编辑 |
| 降水相态 | `ptype_999_revised_YYYYMMDDHH_hhh.txt` | 整数编码，详见第 9 节 |

数据格式：501 行 × 821 列 CSV 纯文本，逗号分隔，3h 步长，000-240h，每日 08Z/20Z 两个起报时次。

### 8.2 编辑主变量

| 变量 | 含义 | 来源 | 是否可编辑 |
|---|---|---|---:|
| qpf_before | 订正前累计降水 | tp 文件差分计算 | 否 |
| ptype_before | 订正前累计相态 | 窗口内 3h ptype 文件实时合成，详见 §8.2.3 | 否 |
| qpf_after | 订正后累计降水 | 初始化为 qpf_before | 是 |
| ptype_after | 订正后累计相态 | 初始化为 ptype_before | 是 |

#### 8.2.1 tp 累积降水差分规则

tp 文件存储的是**从起报时刻起的累积降水量**，获取某一时段的降水量需要对两个时效的 tp 做差分：

```text
qpf_before = tp(end_lead) - tp(start_lead)
```

其中 `end_lead` 和 `start_lead` 分别对应 §7.1 中定义的结束时效和起始时效。

> **参考实现**：`reference/要素场绘图/rain.py:77`，`data = data0 - data1`，其中 data0 对应 end_lead 文件，data1 对应 start_lead 文件。

#### 8.2.2 差分质量控制

后端在计算 `qpf_before` 时，必须执行以下检查：

| 序号 | 场景 | 处理 |
|---:|---|---|
| 1 | `end_lead` 文件不存在 | 该窗口不可用，前端显示"数据缺失"，不进入编辑 |
| 2 | `start_lead` 文件不存在 | 见下方 `start_lead=000` 处理优先级；非 000 时效则窗口不可用 |
| 3 | `qpf_before < 0` 的格点 | 记录 `negative_qpf_mask`，写入日志并在前端标注数据异常；不参与编辑和统计 |
| 4 | `qpf_before` 缺测（NaN） | 对应格点设为无效，不参与编辑和统计 |

`start_lead=000` 处理优先级：

| 优先级 | 条件 | 处理 |
|---:|---|---|
| 1 | `tp_..._000.txt` 文件存在 | 使用实际文件，`qpf_before = tp(end_lead) - tp(000)` |
| 2 | 文件不存在，且配置 `allow_zero_start_lead_fallback=true` | 使用全零数组替代，`qpf_before = tp(end_lead)` |
| 3 | 文件不存在，且配置为 `false` | 窗口不可用 |

> **原因**：部分系统的 000 文件经订正或深度学习处理后可能不是严格全零，默认应优先读取实际文件。配置开关见 §26.2 `product_config.json`。

实现约束：
- 差分结果出现**任何负值格点**时，后端须在窗口元数据中附带以下字段，供前端弹出告警：
  - `negative_count`：负值格点数
  - `negative_min_value`：负值中的最小值（如 -3.6）
  - `negative_abs_max`：负值绝对值最大值（如 3.6）
- 缺测和负值格点在前端渲染时使用斜线阴影填充，与正常降水色标区分。

#### 8.2.3 累计窗口相态生成规则

ptype_before 由后端在加载窗口时实时计算，不依赖预生成文件。

算法：

1. 读取窗口内每个 3h 时效的 ptype 文件
   - 时效范围：`(start_lead, end_lead]`，步长 3h
   - 例：窗口 024-048h → 读取 027, 030, 033, ..., 048 共 8 个文件

2. 计算每个 3h 步长的瞬时降水量用于过滤：
   ```text
   qpf_step[t] = tp[t] - tp[t-3]
   ```
   仅 `qpf_step[t] > 0.1mm` 的时效参与相态统计

3. 逐格点统计有效时效中的相态出现情况：
   - `ptype=1` → `has_rain = true`
   - `ptype=2` → `has_snow = true`
   - `ptype=3` → `has_rain = true, has_snow = true`

4. 合成规则：

   | has_rain | has_snow | 结果 |
   |:---:|:---:|---|
   | √ | × | 1（雨） |
   | × | √ | 2（雪） |
   | √ | √ | 3（雨夹雪） |
   | × | × | 0（无降水） |

边界情况：

| 场景 | 处理 |
|---|---|
| 窗口内某个 3h ptype 文件缺失 | 跳过该时效，不影响其余 |
| 窗口内所有 ptype 文件缺失 | 该窗口不可用 |
| `start_lead=000` | 第一个有效时效为 003 |

> **参考实现**：`reference/cdsz/src/test/java/RainTest.java:113-196`（`getPtypeDatas`），本系统在此基础上增加 qpf 阈值过滤和 ptype=3 处理。

## 9. 相态编码（已确认）

经分析 `reference/cdsz` 中 `GribToPng.pngCombinePtype()` 方法（GribToPng.java:250-261），**已有系统的相态编码**如下：

### 9.1 编码定义

| 编码 | 含义 | 渲染逻辑 | 数据来源 |
|---:|---|---|---|
| 0 | 无降水 | 透明（降水量 <0.1 时 alpha=0） | ptype_999_revised 文件 |
| 1 | 雨 | `getRainRgbaValue()` 绿蓝紫分级 | ptype_999_revised 文件 |
| 2 | 雪 | `getSnowRgbaValue()` 灰色分级 | ptype_999_revised 文件 |
| 3 | 雨夹雪 | `getRainSnowRgbaValue()` 固定粉紫色 | ptype_999_revised 文件 |

> **注意**：此编码与 cdsz 中 `ptypeMap`（站点天气现象：0=晴,1=雨,2=雪,3=少云,4=多云,5=阴）**不是同一套体系**。`ptypeMap` 是站点预报天气现象，包含云量类别；`ptype_999_revised` 是纯降水相态编码，仅涉及降水类型。

> **实际数据现状**：当前样本数据（5 月中旬）出现 0、1、2 三种值。编码 3（雨夹雪）在冬季数据中预期出现。
