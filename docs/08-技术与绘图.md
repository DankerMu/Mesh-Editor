## 18. 技术路线（已确认）

### 18.1 确定架构

```text
前端：Vue 3 + OpenLayers 10.x + TDesign Vue Next + TypeScript
后端：Python FastAPI
格点处理：NumPy / xarray / shapely
绘图服务：Matplotlib + Cartopy（复用已有绘图脚本）
数据格式：NPZ + PNG + JSONL + CSV（原型）→ NetCDF（生产归档）
数据库：SQLite（原型）→ PostgreSQL（生产）
文件存储：本地文件系统，后续可扩展 MinIO
投影坐标系：EPSG:4326（等经纬度投影），前后端、绘图代码统一
部署：Docker（前端 Nginx 容器 + 后端 FastAPI 容器 + 数据卷）
开发流程：本地开发本地测试 → 最终打镜像部署
```

### 18.1.1 参考 WebGL 图层复用策略

参考项目 `yy-webgis-bj` 中包含三种网格渲染方案：

| 图层 | 技术 | 本系统复用方式 |
|---|---|---|
| GirdValueWebGLTileLayer.js | WebGL + DataTileSource | 借鉴框架，**重写颜色映射**为 qpf+ptype 双变量逻辑 |
| CustomTileLayer.js | Canvas + ImageTileSource | 保留为 WebGL 不可用时的 fallback 方案 |
| gridLayer.js | ImageStatic | 底图 PNG 加载沿用 |

**必须修改的部分**：

1. **颜色映射**：GirdValueWebGLTileLayer 按单变量 values/colors 做线性分级。本系统需按 `ptype` 选择色标函数再按 `qpf` 查色级，逻辑参考 `GribToPng.java:pngCombinePtype()`（详见 §10.3.1）。
2. **经度处理**：GirdValueWebGLTileLayer.js 的 `getDataValue()` 包含全球经度归一化 `((lon % 360) + 360) % 360`。固定区域编辑器必须删除此逻辑，改为严格钳位到网格边界。

**可直接复用的部分**：DataTileSource 创建、tile 缓存、WebGL style band 配置、updateData 刷新机制、debug overlay。

### 18.2 选型依据

| 选型 | 依据 |
|---|---|
| Vue 3 + OL | 参考项目 yy-webgis-bj 已验证，含 WebGL 网格瓦片、投影切换、时间轴等可复用组件 |
| TDesign | 参考项目已集成，组件丰富，适合后台管理类界面 |
| FastAPI | 异步高性能，与 NumPy/Matplotlib 生态无缝对接 |
| EPSG:4326 | 已有绘图代码全部使用 PlateCarree，网格数据本身为等经纬度，无需投影转换 |
| Docker | 隔离运行环境（尤其 Matplotlib 字体/Cartopy 地图数据），便于在生产内网部署 |

### 18.3 不建议路线

不建议按 GFIE 的老式重客户端方式直接复刻。原因是：

1. 本项目区域固定、底图固定，Web/轻量前端即可满足。
2. 本项目核心是数据留痕和复盘归档，Web 后端更易做版本、日志、批量任务。
3. 绘图代码已有且为 Python，FastAPI 直接调用最简单。

## 19. 绘图代码接入方案

### 19.1 已有绘图脚本清单

经对 reference/要素场绘图/ 目录完整分析，已有以下绘图能力：

| 脚本 | 功能 | 要素 | 地理范围 |
|---|---|---|---|
| circulation.py | 高空环流形势 | 相对湿度填色 + 位势高度等值线 + 风场 | 70-140E, 15-55N |
| rain.py / rain24.py | 累积降水 | APCP 差分，6 级色标 | 70-115E, 25-50N |
| element.py | 多要素场 | CAPE/TMP_HIGH/SPFH/TCWV/变温/强风 | 70-115E, 25-50N |
| surface.py | 地面形势 | 海平面气压填色 + 10m 风场 | 70-140E, 15-55N |
| cape.py | 对流有效位能 | CAPE 单要素 | 70-115E, 25-50N |
| tmp_high.py | 地面高温 | 2m 温度 | 70-115E, 25-50N |
| poumian1.py | 站点剖面 | 温度/湿度/垂直速度/风/降水 | 57 个站点 |
| app1.py | Flask 后端 | 图片托管服务 | 监听 192.168.2.80:8080 |
| data_utils.py | 数据工具库 | 地图边界读取、GRIB 解析、时间转换 | — |

所有绘图脚本统一使用：
- **投影**：PlateCarree（等经纬度）
- **数据读取**：meteva.base（GDS 格式）和 xarray（GRIB 格式）
- **边界数据**：GeoJSON 国界/省界
- **时间转换**：UTC → Asia/Shanghai (+8h)
- **区域划分**：All / NW / SW / East 四个子区域

### 19.2 已有脚本的问题（不可直接服务化调用）

经逐文件核查，7 个绘图脚本（circulation / rain / rain24 / element / surface / cape / tmp_high）存在以下共性问题，**不能**直接作为 FastAPI 后台任务调用：

| 问题 | 说明 | 影响 |
|---|---|---|
| 全局变量覆盖参数 | 函数签名接收 year/month/day/hour，但内部路径拼接使用脚本级全局变量 `fcst_year`/`fcst_month` 等，参数形同虚设 | 函数不可重入，不可并发 |
| 路径硬编码 | 输入 `/cloud/share-files/SC/mdfs/ECMWF_HR/`、输出 `/gaoli/huishang/static/images/`、边界 `/gaoli/huishang/draw/`、字体 `./simhei.ttf` 全部写死 | 换环境即崩溃 |
| 起报时间自动判断 | 脚本末尾根据 `datetime.now().hour` 判断 08Z/20Z，写死阈值 14 或 16 | 无法按需指定任意起报时次 |
| 批量循环 | 脚本末尾 3-4 层嵌套循环遍历所有层次×时效×区域 | 无法单次调用生成指定一张图 |
| 配置重复 | 省份列表、城市坐标、区域范围、色标在每个脚本中重复定义 | 改一处漏一处 |

> **唯一例外**：`data_utils.py` 是纯工具函数，无全局变量，可直接复用。`poumian1.py` 已采用配置字典模式，结构优于其他脚本，可作为重构参考模板。

### 19.3 接入策略：三层架构

绘图代码接入采用三层架构，将参考脚本的绘图能力提取为纯函数，再由服务层统一调度。

```text
第一层：原始参考脚本（只读，不由业务调用）
reference/要素场绘图/*.py
    └── 保持原样，仅供查阅绘图逻辑和色标定义

第二层：绘图核心函数（纯函数，无全局变量，无 I/O 路径判断）
plotters/
    ├── precip_phase.py      ← 编辑前后降水/相态/差异/掩膜图
    ├── review_synoptic.py   ← 环流形势、地面形势
    ├── review_element.py    ← CAPE/比湿/变温/强风等要素场
    ├── review_profile.py    ← 站点剖面图
    ├── review_composite.py  ← 复盘组合拼图
    └── plot_config.py       ← 统一配置（色标、省界、城市、字体、区域范围）

第三层：绘图任务服务（读 manifest、准备输入、调用 plotter、记录日志）
services/
    └── review_plot_service.py
```

#### 第二层函数签名规范

所有 plotter 函数遵循以下约束：
- **不读全局变量**，所有输入通过参数传入
- **不自行判断当前时间**，由调用方传入 `init_time`
- **不自行拼接路径**，由调用方传入已解析的数据路径或 NumPy 数组
- **返回输出文件路径**，不做后续 I/O

```python
# plotters/precip_phase.py — 编辑产品出图
def plot_precip_product(
    qpf: np.ndarray,           # 501×821 降水数组
    ptype: np.ndarray,         # 501×821 相态数组
    grid_def: dict,            # grid_definition 配置
    product_type: str,         # "before" | "after" | "delta" | "ptype_change" | "touched_mask" | "changed_mask"
    metadata: dict,            # init_time, window_id, version_id 等
    boundary_config: dict,     # 国界/省界路径、省份列表
    output_path: str,
) -> str: ...

# plotters/review_synoptic.py — 环流形势
def plot_circulation_panel(
    hgt_data: np.ndarray,      # 位势高度场
    wind_u: np.ndarray,        # U 风分量
    wind_v: np.ndarray,        # V 风分量
    rh_data: np.ndarray,       # 相对湿度场
    lon: np.ndarray,           # 经度坐标
    lat: np.ndarray,           # 纬度坐标
    level: int,                # 气压层 (hPa)
    init_time: str,            # 起报时间 ISO 格式
    lead_hour: int,            # 预报时效
    region: str,               # "All" | "NW" | "SW" | "East"
    boundary_config: dict,
    output_path: str,
) -> str: ...

# plotters/review_element.py — 要素场
def plot_element_panel(
    data: np.ndarray,          # 要素数据
    lon: np.ndarray,
    lat: np.ndarray,
    element: str,              # "CAPE" | "SPFH" | "TMP_CHANGE" | ...
    init_time: str,
    lead_hour: int,
    region: str,
    boundary_config: dict,
    color_config: dict,        # 该要素的色标配置
    output_path: str,
) -> str: ...
```

#### 第三层服务职责

```python
# services/review_plot_service.py
class ReviewPlotService:
    def generate_product_images(self, window_id, version_id) -> list[str]:
        """编辑保存后，生成 before/after/delta/ptype_change/touched_mask/changed_mask 六张图"""
        ...

    def generate_review_package(self, manifest_path) -> dict:
        """读取 manifest，逐项调用 plotter，记录日志，返回产物清单"""
        ...
```

服务层负责：
1. 从数据库/文件系统读取数组和元数据
2. 解析 manifest 确定需要生成哪些图
3. 调用第二层 plotter 纯函数
4. 记录绘图日志（开始/结束时间、是否成功、错误信息）
5. 返回产物路径列表

#### 重构优先级

| 优先级 | 脚本来源 | 目标 plotter | 说明 |
|---:|---|---|---|
| P0 | rain.py 色标 + 新逻辑 | precip_phase.py | 编辑系统核心出图，首版必须 |
| P1 | circulation.py | review_synoptic.py | 复盘环流形势，一期末 |
| P1 | surface.py | review_synoptic.py | 复盘地面形势，一期末 |
| P1 | element.py / cape.py / tmp_high.py | review_element.py | 复盘要素场，一期末 |
| P2 | poumian1.py | review_profile.py | 复盘剖面图，二期 |
