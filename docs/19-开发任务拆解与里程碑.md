# 19. 开发任务拆解与里程碑

## 19.1 设计目标

本文将完整业务平台拆解为可开发、可排期、可验收的工程任务。每个里程碑都包含输入、输出、涉及模块和验收条件。

---

## M0：工程骨架与基础设施

目标：建立可运行的前后端工程和基础配置。

任务：

```text
M0.1 后端 FastAPI 项目初始化
M0.2 前端 Vue3 + TS + TDesign + OpenLayers 初始化
M0.3 Docker Compose 开发环境
M0.4 数据库模型与迁移框架
M0.5 配置加载 product_config/grid_definition/plot_config
M0.6 统一错误码和响应格式
M0.7 登录/JWT/权限中间件基础
M0.8 PathBuilder 和日志框架
```

验收：

```text
- /api/health 可访问
- /api/auth/login 可返回 token
- 前端可登录并进入空白工作台
- 数据库 migration 可执行
```

## 19.2 并行开发建议

| 小组 | 可并行任务 |
|---|---|
| 数据/后端 A | M1 数据接入、M3 编辑引擎 |
| 后端 B | M0 权限、M4 版本审核发布、M6 配置审计 |
| 前端 | M2 工作台、M4 审核页、M5 复盘中心 |
| 绘图 | M5 plotter 重构、复盘模板 |
| 测试 | M1-M6 测试数据和自动化测试 |

依赖：

```text
M2 依赖 M1 的窗口接口，但可用 mock 先开发。
M3 依赖 M2 的 mask geometry，但可用 fixture 先开发。
M5 依赖 M4 的 version，但 plotter 可独立重构。
```

## 19.4 风险控制

| 风险 | 控制 |
|---|---|
| 绘图脚本重构耗时 | 先完成 precip_phase，其他复盘模板逐步接入 |
| 前端 WebGL 调试复杂 | 保留 Canvas fallback |
| 多 session 并发冲突 | 首版可配置单窗口单 session 锁 |
| 数据样例不足 | M1 前必须准备 normal/partial/invalid/negative 样例 |
| 复盘变量格式不统一 | 先通过 adapter 归一为 np.ndarray + lon/lat |
